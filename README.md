# CS_230_Gaming_Room_project
This repository contains the completed software design document for Draw It or Lose It, developed for The Gaming Room. The project demonstrates object-oriented design, UML interpretation, use of design patterns, and scalable system architecture planning for a web-based multi-platform game application

Client Overview

The Gaming Room partnered with Creative Technology Solutions (CTS) to expand their Android-only game, Draw It or Lose It, into a web-based, multi-platform application 

CS 230 Project 3 Completed

.

The client required a scalable system that would:

Support multiple games

Allow multiple teams per game

Allow multiple players per team

Enforce unique game and team names

Ensure only one GameService instance exists in memory

The goal was to design a flexible, object-oriented system capable of supporting future cloud deployment and distributed users.

What I Did Well

I successfully translated business requirements into a structured domain model aligned with the provided UML. I implemented appropriate design patterns, including:

Singleton pattern for GameService

Iterator-based validation for enforcing uniqueness

The documentation clearly explains architectural decisions and connects them directly to client requirements.

How the Design Document Helped Development

Creating the design document first made development more efficient and structured. It:

Clarified object relationships

Reduced implementation errors

Ensured all requirements were addressed before coding

Provided a roadmap for scalability

This process reinforced the importance of planning before writing code.

What I Would Improve

If revising this project, I would enhance the system architecture section with visual deployment diagrams showing cloud infrastructure, containers, and database tiers. This would strengthen communication with stakeholders and better reflect enterprise-level design standards.

Interpreting User Needs

I focused on scalability, maintainability, and uniqueness constraints when designing the system. By considering future web deployment and multi-user access, I ensured the design aligned with both immediate and long-term business needs.

Understanding user needs is essential because it:

Prevents costly redesigns

Improves usability

Aligns technical solutions with business goals

My Design Approach

My approach included:

Object-oriented design principles

UML modeling

Separation of concerns

Design pattern implementation

Scalability planning

In future projects, I would also incorporate sequence diagrams, API specifications, and security requirement mapping earlier in the design phase.
